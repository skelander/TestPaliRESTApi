Technical Overview

How the Palindrome app is built and deployed.

What it does

A REST API that checks whether a word or phrase is a palindrome — meaning it reads the same forwards and backwards (ignoring spaces and casing). A simple web frontend lets users sign in and use the API, while an admin page controls access per user.

Tech stack

.NET 10 ASP.NET Core xUnit Docker Fly.io GitHub Actions GitHub Pages HTML / CSS / JS

Architecture

The backend is an ASP.NET Core MVC API running on Fly.io. It exposes four controllers:

POST /auth/loginValidates credentials server-side. Returns the username, role, and a signed JWT token (8-hour expiry), or 401 if invalid.
GET /palindrome?input=…Checks if the input is a palindrome and returns a JSON result with input, isPalindrome, and message.
GET /featuresReturns the current API access flag for each user.
PUT /features/{user}Enables or disables API access for a specific user. Requires an admin JWT token.
GET /logsReturns recent application log entries. Requires an admin JWT token.

The frontend is a set of static HTML pages hosted on GitHub Pages. There are no frameworks — just vanilla HTML, CSS and JavaScript calling the API over HTTPS.

Code structure

The backend follows a layered pattern. The controller handles HTTP — it receives the request and returns a response. The service holds the business logic — it knows what a palindrome is. The controller depends only on an interface (IPalindromeService), never the concrete class. This makes it easy to swap implementations and to test each layer independently.

Feature flags are stored in an in-memory dictionary (a ConcurrentDictionary singleton). They are shared across all requests, so a change made by the admin is immediately visible to all users — but they reset if the API restarts.

Testing

There are two layers of tests, both written with xUnit:

Unit testsTest the palindrome logic directly — no HTTP, no framework. Fast and focused on edge cases (phrases, mixed casing, punctuation, nulls).
Integration testsSpin up the full API in-process using WebApplicationFactory and send real HTTP requests. Verify status codes, response bodies and content types.

51 tests in total. A failing test blocks deployment.

CI/CD pipeline

Two independent workflows trigger on push to main, each watching different files:

API changes → Build & Test → Deploy to Fly.io
Frontend changes → Inject API URL → Deploy to GitHub Pages

The pipelines are decoupled — a CSS change only redeploys the frontend, and a code change only runs tests and redeploys the API. The API URL is stored as a repository variable and injected at deploy time, so the frontend contains no hardcoded URLs.

Security note

Login is validated server-side via POST /auth/login. On success the API issues a signed JWT token (HMAC-SHA256). The token is stored in sessionStorage and sent as a Bearer header on every admin request. Protected endpoints return 401 without a valid token.

The JWT secret key is stored as a Fly.io secret and injected at runtime — it is never in the source code. The browser session clears when the tab is closed.
