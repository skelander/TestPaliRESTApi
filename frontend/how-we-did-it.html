How We Did It

This project was built in a live conversation between a developer and an AI coding assistant.

The approach

The entire project was built using Claude Code — an AI assistant made by Anthropic that runs directly in the terminal and IDE. The developer described what they wanted; Claude read the code, made changes, ran tests, and pushed to GitHub. No code was written by hand.

The conversation drove the development. Each step started with a request — sometimes a feature, sometimes a bug report, sometimes just "make the GUI look cooler." Claude proposed an approach, implemented it, and the developer reviewed the result and gave feedback.

What was built, step by step

  • 1Started with a basic palindrome REST API in ASP.NET Core — one endpoint, one service, one test file.
  • 2Added a GitHub Actions CI/CD pipeline: build → test → deploy to Fly.io on every push to main.
  • 3Built a static HTML frontend on GitHub Pages — a palindrome checker that calls the live API.
  • 4Added a login page with test accounts, then moved credentials server-side so they were no longer visible in the frontend source.
  • 5Added per-user feature flags — the admin can enable or disable API access for each user from a toggle panel.
  • 6Replaced hardcoded credentials with an EF Core in-memory database, seeded on startup.
  • 7Fixed a bug: dsdsd%% was incorrectly reported as a palindrome because special characters were being stripped. Changed the logic to only strip whitespace.
  • 8Fixed a second bug: entering . in the GUI caused a network error because a dot in a URL path segment breaks ASP.NET Core routing. Switched the endpoint to use a query parameter instead.
  • 9Split the CI/CD pipeline into two independent workflows — one for the API, one for the frontend — each triggered by the files it cares about.
  • 10Decoupled the frontend from the API: moved HTML files to a frontend/ subfolder and replaced the hardcoded API URL with a GitHub repository variable injected at deploy time.
  • 11Added structured logging to all controllers using ILogger<T>, backed by a custom in-memory log store. The admin page displays a live log viewer that polls GET /logs every 5 seconds.
  • 12Added JWT authentication: login now issues a signed token. All feature and log endpoints require a valid token — admin-only for writes and logs, any authenticated user for reads. The signing key is stored as a Fly.io secret.

What Claude handled autonomously

Beyond writing code, Claude also: ran tests after every change, committed with descriptive messages, pushed to GitHub, watched CI builds and reported back when they completed, kept about.html up to date as the project evolved, and managed the GitHub Actions workflows and repository variables via the gh CLI.
